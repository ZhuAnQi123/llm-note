
# 6.3 业务场景微调 vs RAG vs Prompt

## Q11：【终极架构选型】面对一个全新的企业级业务需求，你如何决定是应该写复杂的 Prompt、搭 RAG 系统，还是直接拉硬件去对模型做参数微调（Fine-tuning）？请给出一套标准的决策树矩阵。

**🎯 核心考点：大模型项目技术选型的架构师眼界与 Trade-off 权衡。**

**参考答案：**
在工业界落地时，为了追求最高效的 ROI（投入产出比），我们通常遵循“Prompt 基线演进 $\to$ RAG 注入知识 $\to$ 微调改变行为”的决策树路径：

1. **第一阶段：优先考量 Prompt Engineering（轻量级验证）**
* *适用条件：* 业务处于 PoC（概念验证）阶段，或者任务通用（如摘要、翻译、简单的格式提取）。
* *特征：* 几分钟就能跑通，零算力成本。如果通过 Few-shot（少样本提示）就能让模型达到 85 分，通常不需要后续的复杂架构。


2. **第二阶段：考量 RAG（当遇到“知识瓶颈”时）**
* *核心触发条件：* 业务高度依赖**企业内部私有数据、长尾动态知识、或者要求数据必须秒级实时更新、且答案必须能够精准追溯出处（防止幻觉）**。
* *决策结论：* 选择 **RAG**。因为微调无法让模型记住实时动态变化的知识（如今天的股票价格、最新的客服公告），微调也无法解决事实溯源问题。


3. **第三阶段：考量 Fine-tuning（当遇到“行为/格式/成本瓶颈”时）**
* *核心触发条件：*
1. **极端严苛的文风与格式控制：** 比如要求模型输出绝对百分之百合法且极其复杂的特殊 DSL 代码。
2. **为了极致的降本增效（Token 瘦身）：** 为了让大模型学会某种长文本业务，我们在 Prompt 里塞了 8000 字的 Few-shot 样例，导致每次调用都极贵且慢。此时，通过微调把这 8000 字的样例直接“烧录”进小模型的参数里，线上运行时 Prompt 只需要 50 个字，实现**用 7B 小模型微调击败 70B 大模型，极大提升 QPS 并压低资费**。
3. **领域极度垂直：** 面对医学、法律等通用模型在预训练时很少见到的生僻专业黑话。


* *决策结论：* 选择 **Fine-tuning**。
