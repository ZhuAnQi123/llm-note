# 3.1 RAG 数据准备与预处理

在工业界，大模型的微调（Fine-tuning）成本高、周期长，且无法实时更新知识。因此，**RAG（Retrieval-Augmented Generation）成为了 90% 以上企业落地 AI 应用的首选方案**。面试官在考 RAG 时，会极度抠工程细节——因为 RAG 的链条极长，任何一个环节没做好，系统就会沦为垃圾场。

-------

### Q1：在 RAG 系统中，为什么说“数据预处理和分块（Chunking）”是决定最终生成效果的上限？如果直接把整篇 5 万字的企业年报喂给 Embedding 模型会发生什么？

**🎯 核心考点：RAG 管道的木桶效应，Embedding 模型的物理限制。**

**参考答案：**

1. **决定上限的原因：** RAG 的核心逻辑是“检索相关知识 -> 喂给大模型”。如果前期的知识库本身被切分得支离破碎、或者混入了大量无用噪声，那么检索出来的上下文（Context）就是有问题的。大模型再聪明，也无法基于错误的材料给出正确的答案。
2. **直接喂 5 万字文本的致命后果：**
* **向量维度稀释与语义模糊：** 每一个 Embedding 模型（如 `text-embedding-3-small`）都有其固定的上下文窗口（通常为 8K Token）。如果强行把几万字压缩进一个向量里，**长文本的全局语义会被极度稀释，局部的核心细节（如某个财务数字）会在空间中彻底丧失特征**，导致检索阶段根本无法精准匹配。
* **物理截断：** 如果文本超出了 Embedding 模型的最大 Token 限制，模型会直接把后半部分硬生生截断扔掉，导致后半张纸的知识永远无法被检索到。
* **大模型窗口与成本暴增：** 即使检索能召回这 5 万字，把它作为 Context 一股脑塞给下游 LLM，也会瞬间吃满大模型的上下文窗口，引发 Lost in the Middle（迷失在中间）现象，并且导致 API 资费暴涨、延迟剧增。

### Q2：请列举至少三种主流的文本分块（Chunking）策略，并说明它们各自适用的业务场景。

**🎯 核心考点：工程落地的 Chunk 划分基本功。**

**参考答案：**
在实际工程中，我们会根据文档的结构特征选择不同的分块策略：

1. **固定大小分块（Fixed-size Chunking）：**
* *机制：* 死板地规定每个块包含固定的字符数或 Token 数（如 `chunk_size=500`），并设置一定的重叠度（`overlap=50`）防止边界语义断裂。
* *场景：* 适用于结构极度松散、无明显段落逻辑的纯文本，或者对计算速度要求极高、预算有限的初级项目。


2. **递归字符分块（Recursive Character Chunking - 最常用）：**
* *机制：* 依照一组分隔符列表（如 `["\n\n", "\n", " ", ""]`）按优先级递归切分。它会尽量保持段落的完整，如果段落太大，再退而求其次切分句子，直到块大小满足要求。
* *场景：* 工业界通用型基线（Baseline）方案。完美适用于小说、新闻报道、常规规章制度等具有明显段落标识的文档。


3. **语义分块（Semantic Chunking）：**
* *机制：* 利用 Embedding 模型动态监测句子之间的语义差异。当连续两句话之间的向量相似度发生剧烈“断崖式下跌”时，说明话题切换了，系统会自动在这里切一刀。
* *场景：* 适用于话题跳转剧烈、逻辑紧密交织的学术论文、深度会议纪要或复杂的商业分析报告。

### Q3：【实战踩坑题】在解析企业资产负债表、产品参数表等带有大量复杂“表格（Table）”的 PDF 文档时，传统的文本提取工具（如 PyPDF, PDFMiner）往往会把表格读得稀碎。请问你在工程上是如何解决 RAG 中表格解析难题的？

**🎯 核心考点：工业界最头疼的“表格检索”实战填坑经验。**

**参考答案：**
传统的 PDF 工具是按物理坐标流抽文本的，会把多栏表格按行错位拼接，彻底破坏表格的二维交叉对应关系。我们在工程上通常采用“多模态/结构化重构”方案来治理：

1. **表格识别与 HTML/Markdown 转化：**
* 摒弃纯文本提取，引入专门的文档智能解析工具（如 Microsoft LayoutLM、PaddleOCR、或开源的 Unstructured / Marker）。
* 强行将 PDF 中的表格区域识别出来，并**转换为标准的 Markdown Table 或 HTML `<table>` 标签文本**。Markdown 的 `| 字段1 | 字段2 |` 结构对 Embedding 模型和大模型都具有极高的语义可读性。


2. **小模型摘要替代法（Summary-to-Chunk）：**
* 对于极度复杂的跨页大表，直接转 Markdown 的效果依然可能由于丢失上下文而变差。
* **工程大招：** 把提取出的表格单独丢给一个多模态模型或高性价比的小 LLM，让它为这个表格生成一段大白话总结（例如：“*本表格是2025年第一季度研发费用明细表，其中A项目投入最高，为500万元……*”）。
* **双路存储：** 将这段“大白话摘要”拿去计算 Embedding 并存入向量库**用作检索；当用户 Query 命中这段摘要时，系统**真正召回并喂给大模型的是原始的 Markdown 完整表格。这完美解决了“向量匹配难”与“大模型看全貌难”的矛盾。

### Q4：在分块时，为什么必须要设置 `Chunk Overlap`（块重叠度）？它的大小通常如何设定？

**🎯 核心考点：解决分块带来的语义断裂问题。**

**参考答案：**

1. **设置 Overlap 的核心原因（防语义切断）：**
语言是有前后连贯性的。如果我们硬生生地在第 500 个字的位置切一刀，可能会把一句话斩成两半（例如前半句在块 A，后半句在块 B）。当用户提问涉及到这句话时，无论单独召回块 A 还是块 B，大模型拿到的都是残缺的上下文，从而引发幻觉。
* **作用：** Overlap 就像是两个滑动窗口之间的“粘合剂”，确保每个 Chunk 的边界区域都保留了一定的前序和后续上下文，维护了语义的连续性。


2. **大小设定调优规范：**
* 通常将 Overlap 设为 `Chunk Size` 的 **$10\% \sim 20\%$**。例如，如果 `Chunk Size` 设为 500 个 Token，`Overlap` 一般设为 `50 ~ 100` 个 Token。
* *业务微调：* 如果文档逻辑密度高（如法律条文，前后条款咬合紧密），应当适当**调大 Overlap**；如果文档是独立的知识点问答对（FAQ 列表），则可以将 Overlap 设为 0。
