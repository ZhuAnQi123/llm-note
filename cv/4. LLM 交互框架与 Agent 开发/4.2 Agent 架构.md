# 4.2 Agent (智能体) 架构

在面试中，Agent 架构是考察“高阶应用设计能力”的重灾区。面试官会通过这一章，判断你是否真正理解大模型是如何从一个“被动响应的文本续写器”，变成一个能够**自我规划、拥有记忆、并能主动操作外部工具**的“独立数字员工”。

---

### Q1：业界公认的 AI Agent 核心架构由哪四大要素组成？请结合一个“自动帮你订机票的 AI 助手”来通俗解释这四大要素的作用。

**🎯 核心考点：Agent 顶层架构设计的标准宏观认知。**

**参考答案：**
根据 AI 领域（特别是 Weng Lilian 等人）的权威定义，一个完整的 Agent 架构可以用公式表达为：**Agent = LLM（大脑） + 规划（Planning） + 记忆（Memory） + 工具（Tools）**。

以“自动订机票的 AI 助手”为例，这四大要素各司其职：

1. **大模型大脑 (Core LLM)：**
* *作用：* 负责中央调度、意图理解与最终决策。它看懂用户说的话（“帮我订一张明天去北京最便宜的机票”），并决定接下来每一步该做什么。


2. **规划能力 (Planning)：**
* *作用：* 负责目标分解与自我反思。面对订机票这个大任务，大脑会将其拆解为子任务：① 查询明天所有去北京的航班；② 过滤出价格最低的；③ 确认扣款信息。如果中途发现没有明天的航班，它还会自我反思（Reflection）并修正路线，主动询问用户要不要改订后天的。


3. **记忆机制 (Memory)：**
* *作用：* 负责保存状态和上下文。
* *短期记忆：* 记住当前对话进行到哪一步了，用户刚刚确认了身份证号。
* *长期记忆：* 记住用户的偏好（例如：该用户一向喜欢坐靠窗的位置、有南方航空的会员卡）。




4. **工具箱 (Tools / Action)：**
* *作用：* 负责拓展 AI 的能力边界（手和脚）。大模型本身不会联网，更没有携程的账号，但它可以通过调用**外部 API**（如机票查询接口、支付网关、短信发送工具）来切实执行现实世界的动作。

### Q2：请详细阐述经典智能体推理架构——ReAct (Reason + Action) 的工作原理。大模型在执行 ReAct 循环时，内部的 Prompt 结构通常是怎样的？

**🎯 核心考点：Agent 动态执行逻辑的底层技术链条。**

**参考答案：**

1. **核心思想：** ReAct 架构的核心是“把‘推理（Reasoning）’和‘行动（Action）’交织在一起”。大模型在面对复杂问题时，先写下自己的“思考过程（Thought）”，然后再决定“采取什么行动（Action）”并执行，接着观察外部环境返回的“结果（Observation）”，如此反复循环，直到解出答案。
2. **标准的 ReAct 状态循环：**
`Thought (思考) -> Action (行动) -> Observation (观察) -> Thought ...`
3. **底层的 Prompt 模板结构示例（面试时说出这个结构极其加分）：**
在工程实现中，ReAct 是通过在 System Prompt 里塞入一套严格的指令协议（Protocol）来强行规范模型的输出格式的：

```text
你是一个具备工具调用能力的 AI 助手。面对用户的问题，你必须严格按照以下格式进行分步思考和执行，绝不能一次性吐出所有结果。

使用以下格式：
【Thought】: 思考你当前面临的问题，并决定下一步需要调用什么工具。
【Action】: 你必须调用且仅调用下方可用工具列表中的一个。格式为：工具名称[参数]。
【Observation】: 这是系统自动返回给你的工具执行结果，你只需阅读它，不能自己伪造。
...（这个 Thought/Action/Observation 的过程可以重复 N 次）
【Final Answer】: 当你通过观察拿到了全部所需信息后，在这里给出回答用户的最终结论。

可用工具列表：
- Calculator[expression]: 数学计算工具。
- WebSearch[query]: 搜索引擎，用于查最新信息。

现在，任务开始：
用户问题：{{ user_query }}

```

### Q3：大模型本身的 API 调用是“无状态”的（即不记得上一轮说了什么）。在构建长链路 Agent 时，你是如何在工程上实现它的“短期记忆”与“长期记忆”的？

**🎯 核心考点：Agent 状态持久化与多级存储架构。**

**参考答案：**
在工程落地中，我们必须通过外部工程框架（如 Redis、向量库）来帮模型“代持”记忆：

1. **短期记忆（Session Memory - 会话级）：**
* *原理：* 负责维持单次对话的连贯性。
* *工程实现：* 将每一次的用户输入（User）和模型返回（Assistant）按时间顺序保存到高性能的内存数据库（如 **Redis** 或持久化 DB）中。
* *动态修剪（最关键）：* 随着对话变长，为了防止吃满大模型的 Token 窗口，系统会在后端进行**滑动窗口裁剪（Windowing）**、或者使用一个小模型对前 10 轮的对话自动生成一段**摘要（Summary Memory）**，只把摘要和最近 2 轮的对话当成短期记忆拼进当前 Prompt。


2. **长期记忆（Semantic Memory - 跨会话级）：**
* *原理：* 负责记住用户的习惯、个人画像、或者几个月前的核心事实。
* *工程实现：* 当 Agent 发现用户提到了关键个人特征（如“我对花生过敏”）时，后端会自动抽取这个知识点，计算其 Embedding 向量，存入**向量数据库（Vector DB）**。
* *召回机制：* 两个月后，用户再次提问：“帮我推荐一家今晚聚餐的餐厅。”系统会在检索阶段去向量库里默默召回一条长期记忆：“用户对花生过敏”，并将其作为约束喂给大模型大脑。

### Q4：在 Agent 执行 Tools 阶段，经常由于模型的随机性导致生成的工具参数格式错误（例如把 JSON 写歪了），或者传入了非法的参数。你有哪些工程手段来提升工具调用的鲁棒性？

**🎯 核心考点：Agent 线上落地的稳定性防御与容错机制。**

**参考答案：**
工具调用（Tool Calling / Function Calling）失败是 Agent 线上最常见的 Crash 原因。我在实际项目中使用以下**四层防御策略**来彻底控死报错率：

1. **原生 Function Calling 能力压制（优先选型）：**
摒弃纯文本 Prompt 盲猜，调用模型厂商（如 OpenAI、DeepSeek）原生支持的 `tools` 参数，并在 API 级别将 `tool_choice` 设为 `required` 或指定工具。这样模型在解码层就会被强行限制，输出的必然是符合 JSON Schema 的参数，失败率直接骤降。
2. **基于 Pydantic 的强类型校验：**
在后端代码中，为每一个 API 工具定义一个 Pydantic 的 `BaseModel` 结构体。模型生成的参数进入工具前，必须通过 Pydantic 校验。如果缺字段或类型不对（如字符串传成了布尔值），直接在代码层拦截。
3. **自动错误回溯与自我修复（Self-Correction Loop）：**
如果 Pydantic 校验报错，不要向用户报错。**把具体的报错信息（Traceback）直接当成 Observation 喂回给大模型**，对它说：`“你刚才调用的工具参数报错了，错误信息是：XXXX。请根据报错重新思考（Thought）并修正参数再次调用。”`大模型通常能在第 2 轮内自我修复成功。
4. **严格的超时与沙箱隔离：**
Agent 调用的外部工具代码必须设置强超时时间（如 5 秒），防止由于外部 API 卡死导致整个 Agent 线程夯住；执行动态代码（如 Python 代码解释器）必须放在 Docker 隔离沙箱中运行，防止恶意的安全越狱和删库命令。
