# 4.1 主流开发框架

如果说 RAG 是给大模型配了一部“查阅资料的字典”，那么 **Agent（智能体）** 就是给大模型安上了“大脑的思考回路和一双能干活的手”。面试官在这一章非常喜欢考察你的**工程反思能力**（如对 LangChain 的祛魅）以及对**复杂动态链路（如 ReAct）的掌控力**。


---

### Q1：请简述 LangChain 框架的核心设计思想。它提供了哪些核心组件的抽象？

**🎯 核心考点：主流大模型开发框架的基础认知。**

**参考答案：**
LangChain 的核心设计思想是“通过将大模型（LLM）与外部数据源、计算逻辑及工具链进行‘链式（Chain）’组合，来构建复杂的上下文感知与推理应用”。

它主要提供了以下 6 大核心组件的规范化抽象：

1. **Models（模型接入）：** 抹平不同厂商 API 的差异。无论是 OpenAI 的 ChatOpenAI 还是开源的 HuggingFaceLocal，都抽象出统一的 `.invoke()` 或 `.stream()` 接口。
2. **Prompts（提示词模板）：** 提供参数化的 `PromptTemplate` 和 `ChatPromptTemplate`，支持角色（System/User）隔离与动态数据拼装。
3. **Indexes / Vector Stores（数据检索）：** 统一了文档加载器（Document Loaders）、文本分块器（Text Splitters）以及各家向量数据库的连接器。
4. **Chains（链）：** 将多个组件串联的核心逻辑。在现代 LangChain 中，通过 **LCEL（LangChain Expression Language，表达式语言）** 利用管道符 `|` 实现声明式的链式调用（如 `prompt | model | parser`）。
5. **Memory（记忆）：** 在无状态的 HTTP API 之上，抽象出自动管理历史对话上下文的组件（如 `ConversationBufferMemory`）。
6. **Agents / Tools（智能体与工具）：** 允许大模型自主决定调用哪些外部函数（Tools），并提供动态执行循环的运行时环境。

### Q2：【高频尖锐问】LangChain 在大模型刚爆发时红极一时，但为什么现在许多大厂、一线架构师以及资深 AI 工程师都在高呼“抛弃 LangChain，重回原生开发或选择轻量级框架”？它在生产落地中存在哪些一致公认的致命痛点？

**🎯 核心考点：具备独立思考与填坑经验的架构师视角。**

**参考答案：**
这几乎是目前中高级大模型工程开发面试必问的经典问题。LangChain 确实存在严重的“过度设计（Over-engineering）”，在线上高并发、长链路的生产环境下暴露出以下致命痛点：

1. **过度抽象导致黑盒化，Debug 极度痛苦：**
LangChain 把底层的 API 和 Prompt 封装得太深了。当线上模型返回不符合预期的垃圾 JSON 时，开发者很难看清到底是 LangChain 内部偷偷加了什么 System Prompt，还是被它自带的后处理解析器截断了。为了改一个微小的提示词，经常需要层层重写它内部的 Base 类，代价极大。
2. **LCEL 陡峭的学习曲线与魔改困难：**
LangChain 强推的 LCEL 表达式语言（如 `chain = prompt | model`）虽然看起来优雅，但在 Python 原生逻辑里创造了一套高特异性的特殊语法。一旦业务需要加入复杂的动态分支判断、条件重试、或者与传统业务系统的微服务进行深度交织时，LCEL 的拼装会变得异常恶心和脆弱。
3. **升级过于激进，API 稳定性极差：**
大模型技术迭代快，LangChain 的版本迭代极其混乱，经常在小版本更新中直接废弃（Deprecated）核心类，导致企业线上代码在升级依赖时频繁遭遇不兼容崩溃（Breaking Changes）。
4. **总结与大厂选型趋势：** LangChain 极其适合做**快速的原型 Demo 验证（PoC）**。但在大厂核心业务落地时，为了追求**高并发吞吐、绝对的确定性控制、以及极低的延迟**，架构师更倾向于自己用 Python 的 `boto3` / `openai` 官方库进行轻量级原生封装，或者选择像 **Langfuse / Graph-based 框架**（如微软的 Semantic Kernel，或专注于状态机的 LangGraph）来掌握代码的绝对控制权。

### Q3：什么是 LlamaIndex？它和 LangChain 的侧重点有什么不同？在什么业务场景下你会优先选择 LlamaIndex？

**🎯 核心考点：工具链矩阵对比与技术选型眼界。**

**参考答案：**

1. **核心不同点与侧重点：**
* **LangChain 侧重于“通用型工作流组装（Workflow）”**：它是一个万金油框架，关注的是模型、链条、Agent 工具调用等全方位的逻辑编排。
* **LlamaIndex 侧重于“数据管道与深入检索（Data Stack for RAG）”**：它的前身是 GPT Index，所有的核心设计全部围绕着“如何把复杂的企业异构数据（SQL、NoSQL、PDF、Notion）高效转化为大模型可理解的索引结构”。


2. **架构设计差异：**
LlamaIndex 提供了极其高级且强大的数据图结构抽象（如 `Property Graph`、`Tree Index`）。它对文档分块、高级语义召回、多路节点路由等 RAG 精细操作的封装质量，远远超过 LangChain 里的 VectorStore 模块。
3. **技术选型场景：**
* 如果公司的核心需求是**构建一个企业级知识库、一个需要处理复杂多表关联/多文档关联的深度 RAG 系统**，我会毫不犹豫地**优先选择 LlamaIndex**。
* 如果公司的需求是**构建一个需要调用各种外部 API、有复杂人机交互流程、包含多 Agent 协同的自动化流程工具**，则 **LangChain 或原生开发**会更加合适。
