# 1.2 高级提示策略
是面试中含金量最高、也是最容易拉开差距的部分。面试官非常喜欢在这里考察你**解决复杂逻辑、数学推理、或者长链路任务的实战经验**。

这里不考简单的“换个词”，考的是“如何像训练团队一样去操控大模型的思考路径”。

# 📝 LLM 面试题库专栏：1.2 高级提示策略 (Advanced Strategies)

本文件包含 6 道大厂高频面试真题，涵盖 Few-shot 的底层坑、CoT 的演进路线、以及大名鼎鼎的 **DSPy（自动化提示词工程）**。

---

### Q1：什么是思维链（CoT, Chain of Thought）？为什么“Let's think step by step”能显著提升大模型的推理能力？其背后的数学/原理是什么？

**🎯 核心考点：CoT 底层原理，大模型生成机制的本质。**

**参考答案：**

1. **定义：** 思维链（CoT）是一种通过引导大模型在输出最终答案之前，**先生成一系列中间推理步骤**的提示策略。
2. **为什么有用（底层原理）：**
* **Token 级算力分配（自回归机制）：** 大模型是自回归的，每次只能根据之前的所有 Token 预测下一个 Token。如果直接要答案，模型只有一次计算机会（概率单步解码）。而“一步步思考”迫使模型把一个复杂的推理过程拆解成多个简单的中间 Token，**每一次输出中间步骤，都是在为生成最终答案“积累正确的上下文算力”**。
* **降低条件熵：** 复杂的数学题或逻辑题，直接从题目推导到答案的条件概率非常低。通过中间步骤的拆解，将大任务变成多个高概率的连续小任务，从而显著提高最终答案的准确率。

### Q2：在小样本提示（Few-shot Prompting）中，仅仅提供几个例子就行了吗？样本的选择和顺序对模型输出会有什么影响？线上环境如何避免“样本偏见”？

**🎯 核心考点：Few-shot 的隐形坑与鲁棒性优化。**

**参考答案：**
Few-shot 绝不是随便放几个例子那么简单。学术界和大厂工程实践表明，LLM 对 Few-shot 极其敏感，存在严重的“多数偏见”**和**“近因效应”。

1. **影响因素：**
* **样本标签分布（多数偏见）：** 如果给模型的 4 个例子里，有 3 个是“正面评价”，1 个是“负面评价”，模型会倾向于频繁输出“正面评价”（即使新输入明显是负面的）。
* **样本顺序（近因效应）：** 模型更容易模仿**最靠近 Prompt 末尾**的那个例子的格式和倾向。
* **样本相似度：** 如果 Few-shot 例子与当前用户输入的话题风马牛不相及，模型可能只会死板地模仿格式，而无法准确推理。


2. **线上避坑解决方案（如何变高级）：**
* **标签平衡：** 确保示例中各类标签的数量绝对对等。
* **动态少样本（Dynamic Few-shot）：** 线上不要用死代码写死几个例子。应当使用 **Embedding 向量检索**，根据用户当前输入的 Query，去“示例库”里动态召回 3 个最相似的例子拼进 Prompt。
* **格式一致性：** 严格统一示例的输入输出边界（如使用统一的 XML 标签括起来）。

### Q3：从零样本（Zero-shot）、思维链（CoT），到自洽性采样（Self-Consistency），再到思维树/图（ToT / GoT），这些提示策略的演进逻辑是什么？请对比它们的优缺点。

**🎯 核心考点：高级提示策略的技术演进脉络，架构选型能力。**

**参考答案：**
这一系列策略的演进，本质上是“将模型的思考链路从‘线性单路径’向‘非线性多路径/搜索树’演进”。

| 策略 | 核心思想 | 优点 | 缺点 / 业务痛点 |
| --- | --- | --- | --- |
| **Zero-shot** | 直接输入指令要答案。 | 延迟极低，最省 Token。 | 无法处理复杂逻辑和多步计算。 |
| **CoT** | 线性推理（给出或诱导中间步骤）。 | 显著提升逻辑推理和计算能力。 | 依然是单次生成，如果中间某一步算错，后面全满盘皆输（容错率低）。 |
| **Self-Consistency** | 多路并行 CoT + 投票（设置 Temperature 大于 0 采样多次，取出现频率最高的答案）。 | **大幅提升数学/代码题的准确率**，容错率极高，能有效抹平单次生成的偶然错误。 | **成倍增加 Token 成本和延迟**。如果调用 5 次，资费和耗时就是 5 倍。 |
| **ToT / GoT (树/图)** | 引入回溯、分支、评估机制（如 A*搜索、DFS），让模型生成多个方案并自我打分，不好就回溯。 | 理论上逼近人类攻克奥数、复杂调度等顶尖难题的能力。 | 结构极其复杂，需要大量的工程代码（如 Python 控制循环）配合，**极慢且极贵**，极少直接用于高并发线上 C 端。 |

### Q4：在某些复杂分类任务中，模型往往“想得太多”或者“抓不住重点”，听说过方向性刺激提示（Directional Stimulus Prompting）吗？它是怎么解决这个问题的？

**🎯 核心考点：长文本摘要或复杂属性提取中的精细化引导。**

**参考答案：**

1. **定义：** 方向性刺激提示（DSP）是指在给模型的输入中，除了核心指令和上下文外，额外注入一个“微小的暗示或关键词向导（Stimulus）”，用来给模型的注意力机制（Attention）“指路”。
2. **应用场景：** 例如让大模型根据一封长邮件写摘要，如果直接让它写，它可能会总结一些无关紧要的客套话。
3. **实现方式：**
* *普通 Prompt：* “请总结以下文章。”
* *DSP Prompt：* “请总结以下文章。**【提示关键词：公司收购、融资金额】**”
* 这种“Hint（提示点）”就像是给模型画了重点，在不需要提供长样本（Few-shot）的情况下，能极大地将模型生成的 Token 概率锚定在特定主题上，既省 Token 又精准。

### Q5：【高阶必问】现代提示词工程正在告别“人工手写（Hand-crafted）”，进入自动化时代。请谈谈你对 APE（自动提示工程）或者斯坦福 DSPy 框架的理解。

**🎯 核心考点：提示词工程的未来趋势，区分低端调参侠与高阶工程专家。**

**参考答案：**

1. **传统手写 Prompt 的痛点：** 极度依赖“玄学和直觉”；只要底层模型一升级（比如从 GPT-3.5 换到 GPT-4，或者换成开源的 Llama），原本调好的 Prompt 就会全面失效（也就是 Prompt 极其缺乏**可迁移性**）。
2. **APE (Automatic Prompt Engineer) 原理：** 利用“模型生成模型”。让一个大模型（LLM1）针对给定的输入输出数据集，自动生成 10 个不同的候选 Prompt，然后用另一个大模型（LLM2）或者评测集去跑分，挑选胜出者。
3. **DSPy 框架的革新（核心重头戏）：**
* **从“编写提示词”转变为“编写程序”：** DSPy（Demonstrate-Search-Predict）彻底抛弃了在字符串里写 `Let's think step by step` 的传统做法。它模仿了 PyTorch 的现代软件工程结构。
* **核心机制：** 开发者只需定义 **Signatures**（输入是什么，输出是什么，比如 `question -> answer`）和 **Modules**（如 `Predict` 或 `ChainOfThought`）。
* **自动编译优化（Compiler）：** 只要你给它提供几十个训练样本，DSPy 的编译器就会**自动去尝试、优化、并生成最适合当前底层模型的 Few-shot 示例和提示词指令**。如果底层模型从 OpenAI 换成了 DeepSeek，你不需要改代码，只需重新运行一下 `dspy.teleprompt` 编译，它就会自动为你适配新模型的最佳提示策略。
