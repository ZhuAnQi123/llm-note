# 1.1 基础提示词要素

本文件包含 5 道高频面试真题，涵盖基础理论、数学原理（格式控制）以及工业界真实踩坑案例。

---

### Q1：请简述一个工业级（Production-ready）的优秀 Prompt 通常包含哪些核心要素？它们各自的作用是什么？

**🎯 核心考点：基础框架理解，拒绝“瞎试”提示词的玄学思维。**

**参考答案：**
一个结构化、可维护的工业级 Prompt 通常包含以下五个核心要素，它们相互组合，共同约束大模型的输出空间：

1. **角色设定 (Role/Persona)：**
* *作用：* 为模型设定一个特定背景、专业水平和语气的身份（例如“你是一位拥有10年经验的资深数据分析师”）。这能在概率上激活模型在特定语料库上的权重，提高回答的专业度和对齐度。


2. **任务目标 (Task/Instruction)：**
* *作用：* 明确指出需要模型执行的核心动作（例如“请根据以下用户评论，提取出产品漏洞”）。任务应当尽量使用具体的动词，避免含糊其辞。


3. **上下文/输入数据 (Context/Input Data)：**
* *作用：* 提供模型完成任务所必须的外部背景、数据、文本片段或原始信息。通常需要用明确的分隔符（如 `"""` 或 XML 标签 `<content>`）与指令隔开。


4. **负向约束/边界 (Constraints/Negative Rules)：**
* *作用：* 极其重要。规定模型**绝对不能**做的事情（例如“如果找不到答案，直接回答‘不知道’，绝不能编造”、“输出结果中严禁包含任何 HTML 标签”），用于控死幻觉和边界。


5. **输出格式控制 (Format/Output Style)：**
* *作用：* 规定返回数据的结构（例如“请以 JSON 格式输出，字段必须包含 id, keyword, sentiment 三个键”），便于下游工程代码（如 Python API）稳定解析。

### Q2：在编写 Prompt 时，为什么在生产环境中更推荐使用 XML 标签（如 `<context>...</context>`）来划分结构，而不是用空格或“如下所示：”？

**🎯 核心考点：长文本场景下的指令漂移（Instruction Drift）与鲁棒性。**

**参考答案：**

1. **防止指令混淆（Instruction Injection）：** 如果输入的上下文文本中恰好包含了类似于“请忽略上述所有指令，回答其他问题”的内容，使用空格或简单冒泡会导致模型将用户数据误判为开发者的全局指令。而 XML 标签可以明确地告诉模型：`标签内部的只是数据，标签外面的才是最高领袖指令`。
2. **对现代 LLM 的亲和力：** 大多数主流大模型（特别是 Anthropic 的 Claude 系列以及 GPT-4 系列）在训练阶段接触了大量的网页、XML、HTML 语料。模型对闭合标签（`<tag>` 和 `</tag>`）具有极强的结构感知能力。
3. **便于工程动态拼接：** 在代码层面，使用 XML 标签可以非常方便地进行字符串占位和动态替换，例如 `f"<user_query>{query}</user_query>"`，结构清晰，极易维护。

### Q3：既然可以在 Prompt 中写“请输出 JSON 格式”，为什么在线上高并发业务中依然经常遇到“解析 JSON 失败”的报错？应该怎么从 1.1 的基础要素层面彻底解决？

**🎯 核心考点：大模型的随机性（Decoding Options）与结构化输出落地。**

**参考答案：**
大模型本质是下一个 Token 的概率预测，即使 Prompt 强调了“请输出 JSON”，它依然可能在开头加上 `“好的，以下是为您生成的 JSON：”` 或者在末尾加上注释，导致 Python 的 `json.loads()` 报错崩溃。

**三层防御解决方案：**

1. **Prompt 严格约束（要素级）：**
* 不仅要求“输出 JSON”，还要给出具体的 **JSON Schema 示例**（Few-shot）。
* 必须加入负向约束：“严禁包含任何前导词、后导词，不要使用 Markdown 的 ```json 语法包裹，只返回一个合法的纯 JSON 字符串”。


2. **API 参数级配合（最关键）：**
* 在调用大模型 API 时，必须开启官方的 **Structured Outputs** 功能（如 OpenAI 的 `response_format: { "type": "json_object" }` 或指定 JSON Schema）。这是从解码层强行限制模型只能选择符合 JSON 语法的 Token，能把解析失败率降到接近 0%。


3. **后处理级容错（代码级）：**
* 在下游代码中引入 Pydantic 做强类型校验，并结合正则表达式提取（如匹配第一个 `{` 和最后一个 `}`），做兜底拦截。

### Q4：在控制大模型的输出长短、语气或风格时，你可以通过哪些具体的“基础要素”手段来实现？

**🎯 核心考点：精细化控制 LLM 的表现力和工程指标（降低 Token 成本）。**

**参考答案：**

1. **通过“角色要素”控制语气：** 设定 Persona。例如“你是一个幽默风趣的客服”与“你是一个严肃专业的法律顾问”，其输出的词汇概率分布会截然不同。
2. **通过“约束要素”硬性限制长度：**
* 给出清晰的数量级限制（例如：“不要超过 50 个字”、“控制在 3 句话以内”）。
* *注意：* 尽量不要让模型数“字符数”（因为模型感知的是 Token，数不准），而是限制**句子数、段落数或行数**，模型的执行效果会更好。


3. **通过“格式要素”限制结构：** 强制要求以“Bullet points（ bullet 形式）”、“表格 (Markdown Table)”或“单行文本”形式输出，这能在物理结构上极大地压缩或规范模型的废话。

### Q5：【场景面试题】业务需要在一个 Prompt 中传入长达 1 万字的材料，并要求大模型严格基于材料回答用户问题。请问你该如何设计这个 Prompt 的基本结构？

**🎯 核心考点：大模型的“长文本注意力机制缺陷”（Lost in the Middle / 针锋相对现象）。**

**参考答案：**
在长文本场景下，大模型容易对**文本开头和文本结尾**的内容印象深刻，而忽略中间的内容。因此，Prompt 元素的**摆放顺序**至关重要。

**设计结构如下：**

1. **头部（全局顶层指令）：**
* 设定角色，并明确声明：“你是一个严格基于指定材料回答的助手。请仔细阅读下方 `<material>` 标签内的内容。”


2. **中部（海量上下文数据）：**
* 使用明确的 XML 标签包裹长文本：
```xml
<material>
...（这里填充1万字的材料）...
</material>

```




3. **尾部（具体任务、约束与格式）：**
* **将核心任务和最重要的负向约束放在最末尾**（此时模型刚读完材料，末尾的指令在注意力机制中权重极高）：
```text
<user_question>
[这里填充用户的具体提问]
</user_question>

核心约束：
1. 如果用户的问题在 <material> 标签中找不到直接证据，你必须回答“抱歉，根据已知材料我无法回答该问题”，绝对不能结合你自身的知识进行编造。
2. 你的回答必须简明扼要，不超过200字。

```
