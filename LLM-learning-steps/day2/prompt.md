

# GPT-5.4 提示词工程指南

**适用场景**：面向长时任务、工具调用与可靠执行。

**文档目标**：本指南旨在帮助开发者掌握针对 GPT-5.4 模型的高效提示词模式与迁移实践，以充分发挥其在复杂工作流中的性能优势。

---

# 一、理解 GPT-5.4 的行为特性

## 1. 核心优势领域
GPT-5.4 在以下方面表现尤为出色：

-   **强一致性与低漂移**：能更好地遵循设定的个性和语气，在长文本生成中保持稳定。
-   **Agentic 工作流稳健性**：更倾向于坚持多步骤任务、自动重试并完成代理循环。
-   **基于证据的综合**：尤其在长上下文或多工具工作流中，能生成基于充分证据的结论。
-   **指令遵循**：在模块化、基于技能和块结构的提示词中，若任务契约明确，遵循效果极佳。
-   **长上下文分析**：能有效处理庞大、混乱或多文档的输入。
-   **批量/并行工具调用**：在保持调用准确性的同时，支持高效的工具调用。
-   **电子表格与金融工作流**：在需要指令遵循、格式保真度和自我验证的任务（如 Excel）中表现出色。

## 2. 仍需明确的提示方向
即便有上述优势，在以下场景中，**更明确的引导**仍然至关重要：

-   **对话初期工具路由**：在上下文信息不足时，模型选择工具的可靠性会降低。
-   **依赖感知工作流**：需要明确检查前置条件和下游步骤的任务。
-   **推理强度选择**：并非越高越好，需根据任务形状选择，而非凭直觉。
-   **研究任务**：要求严谨的信息来源收集和一致的引用格式。
-   **不可逆或高风险操作**：执行前必须进行验证。
-   **终端或编码代理环境**：必须清晰界定工具使用边界。

> **建议**：从能通过评估的最小提示词开始，仅在修复特定失败模式时添加额外指令块。

# 二、核心提示词模式

| 指令块名称 | 作用介绍 | 适用场景 | 是否需要判断场景 | 推荐程度 |
|-----------|---------|---------|-----------------|---------|
| **output_contract** | 强制模型按指定格式输出（JSON/Markdown/SQL等），不添加额外文字 | 所有需要结构化输出的场景 |  ✅必加三件套 | ⭐⭐⭐⭐⭐ |
| **verbosity_controls** | 控制回答简洁度，禁止啰嗦和重复 | 所有场景，尤其是需要快速响应的场景 |  ✅必加三件套 | ⭐⭐⭐⭐⭐ |
| **grounding_rules** | 强制模型基于事实回答，禁止编造，不确定就说不知道 | 所有需要准确性的场景 |  ✅必加三件套 | ⭐⭐⭐⭐⭐ |
| **instruction_priority** | 定义指令优先级：用户新指令覆盖旧指令，但安全规则不可覆盖 | 多轮对话、用户可能中途改变主意的场景 | ⚠️ 可选，客服类推荐加 | ⭐⭐⭐⭐ |
| **default_follow_through** | 定义何时直接执行、何时需要询问用户（低风险直接做，高风险要确认） | 客服机器人、自动化Agent | ⚠️ 可选，有自动化动作时加 | ⭐⭐⭐⭐ |
| **tool_persistence_rules** | 强制模型持续调用工具直到任务完成，不中途放弃 | 多步骤检索、依赖工具完成的任务 | ✅ 需要判断（仅当任务需要工具时） | ⭐⭐⭐⭐ |
| **dependency_checks** | 强制检查前置步骤，不跳过依赖项 | 有先后依赖关系的多步骤任务 | ✅ 需要判断 | ⭐⭐⭐ |
| **parallel_tool_calling** | 指导模型何时并行调用工具、何时串行 | 需要多源检索、追求响应速度的场景 | ✅ 需要判断 | ⭐⭐⭐ |
| **completeness_contract** | 强制覆盖所有要求项，未完成前不结束 | 批量处理、列表遍历、多文档分析 | ✅ 需要判断（长任务推荐） | ⭐⭐⭐⭐ |
| **empty_result_recovery** | 当工具返回空结果时，自动尝试备用策略 | 检索、搜索类任务 | ✅ 需要判断 | ⭐⭐⭐ |
| **verification_loop** | 最终输出前进行正确性、格式、安全性验证 | 高风险操作、代码生成、重要输出 | ⚠️ 可选，推荐用于重要任务 | ⭐⭐⭐⭐ |
| **missing_context_gating** | 信息不足时禁止猜测，要求澄清或使用工具 | 需要精确信息的场景 |  不需要，✅推荐常加 | ⭐⭐⭐⭐ |
| **action_safety** | 执行操作前总结意图，执行后确认结果 | 有副作用的操作（发送、删除、购买等） | ✅ 需要判断（仅高风险操作） | ⭐⭐⭐⭐ |
| **citation_rules** | 强制附上引用来源，禁止编造引用 | 研究、分析、需要可追溯性的场景 | ✅ 需要判断 | ⭐⭐⭐⭐⭐ |
| **research_mode** | 三阶段研究：计划→检索→综合，深度分析 | 研究报告、竞品分析、多文档综合 | ✅ 需要判断 | ⭐⭐⭐⭐ |
| **structured_output_contract** | 针对解析敏感格式（SQL/JSON）的严格输出约束 | 数据库查询、API集成 | ✅ 需要判断（仅格式敏感任务） | ⭐⭐⭐⭐ |
| **bbox_extraction_spec** | 提取文档中的坐标框（OCR/版面分析） | 文档解析、图像文字定位 | ✅ 需要判断（仅视觉任务） | ⭐⭐⭐ |
| **autonomy_and_persistence** | 强制模型端到端完成任务，不中途停止，主动解决问题 | 编码Agent、自动化任务执行 | ✅ 需要判断（仅编码/自动化） | ⭐⭐⭐⭐ |
| **user_updates_spec** | 定义向用户汇报进度的频率和格式 | 长时间运行的任务、编码Agent | ✅ 需要判断（仅长任务） | ⭐⭐⭐ |
| **personality_and_writing_controls** | 定义人物角色、语气、渠道、长度 | 客服、内容创作、邮件撰写 | ✅ 需要判断（需要特定风格时） | ⭐⭐⭐⭐ |
| **memo_mode** | 专业备忘录风格：精确、综合、有结论 | 法律、政策、研究报告、高管汇报 | ✅ 需要判断 | ⭐⭐⭐ |
| **frontend_tasks** | 前端设计规范：布局、品牌、卡片、动效 | AI生成网页/UI代码 | ✅ 需要判断（仅前端任务） | ⭐⭐⭐ |
| **terminal_tool_hygiene** | 终端操作规范：正确使用shell命令，不混用 | 编码Agent、DevOps自动化 | ✅ 需要判断（仅终端操作） | ⭐⭐⭐ |
| **dig_deeper_nudge** | 鼓励模型不满足于第一个答案，寻找深层问题 | 安全审查、质量检查、深度分析 | ⚠️ 可选，推荐用于重要任务 | ⭐⭐⭐ |

## 怎么使用

### 永远可以加的（全局有效）

```yaml
必加三件套:
  - output_contract      # 控制输出格式
  - verbosity_controls   # 控制回答简洁度  
  - grounding_rules      # 确保事实准确性

效果: 让所有回答更规范、更简洁、更可信
成本: 约200-300 token，几乎可忽略
```

### 按场景选择

| 你的场景 | 推荐添加的指令块 |
|---------|-----------------|
| 简单问答 | 必加三件套 + instruction_priority |
| 客服机器人 | 必加三件套 + instruction_priority + default_follow_through + personality_controls |
| 研究分析 | 必加三件套 + research_mode + citation_rules + completeness_contract |
| 编码Agent | 必加三件套 + autonomy_and_persistence + verification_loop + tool_persistence |
| 多步工具调用 | 必加三件套 + tool_persistence + dependency_checks + completeness |
| 高风险操作 | 必加三件套 + verification_loop + action_safety + default_follow_through |
| 前端开发 | 必加三件套 + frontend_tasks + verification_loop |
| 专业写作 | 必加三件套 + memo_mode + personality_controls |
| 批量处理 | 必加三件套 + completeness_contract + empty_result_recovery |

## 提示词详情

### 1. 保持输出紧凑与结构化
为提升令牌效率，通过清晰的**输出契约**约束内容冗长，并强制结构化输出。

```xml
<output_contract>
- 严格按请求的顺序和章节返回内容。
- 若提示词定义了前言、分析块或工作区，不要将其视为额外输出。
- 长度限制仅应用于其指定的章节。
- 若要求特定格式（JSON, Markdown, SQL, XML），则只输出该格式。
</output_contract>

<verbosity_controls>
- 偏好简洁、信息密集的写作风格。
- 避免重复用户请求。
- 进度更新保持简短。
- 不要为了缩短而省略必要的证据、推理或完成度检查。
</verbosity_controls>
```

### 2. 设定清晰的跟进默认行为
定义明确的规则，告知模型何时自行推进、何时询问用户，以及新指令如何覆盖旧指令。

```xml
<default_follow_through_policy>
- 若用户意图清晰且下一步是可逆、低风险的，直接执行，无需询问。
- 仅在以下情况请求许可：
  (a) 操作不可逆，
  (b) 有外部副作用（如发送、购买、删除或写入生产环境），
  (c) 需要缺失的敏感信息或会显著改变结果的选择。
- 若直接执行，简要说明你做了什么以及哪些步骤仍是可选的。
</default_follow_through_policy>
```

```xml
<instruction_priority>
- 用户指令优先于默认的风格、语气、格式和主动性偏好。
- 安全、诚实、隐私和权限限制不可逾越。
- 若新的用户指令与旧指令冲突，遵循新指令。
- 保留不冲突的旧指令。
</instruction_priority>
```

### 3. 处理对话中的指令更新
当任务中途变更时，使用明确、限定范围的引导消息，说明范围、覆盖内容和保留内容。

```xml
<task_update>
任务已变更。
先前任务：完成工作流。
当前任务：仅审查工作流并识别风险。

本轮规则：
- 不执行任何操作。
- 不调用破坏性工具。
- 精确返回：
  1. 主要风险
  2. 缺失信息
  3. 建议的下一步
</task_update>
```

### 4. 确保工具调用的持久性与正确性
对于依赖先前检索或验证步骤的工作流，使用显式规则，避免跳过前置步骤。

```xml
<tool_persistence_rules>
- 只要工具能显著提高正确性、完整性或事实依据，就应使用。
- 在能显著提高正确性或完整性的情况下，不要过早停止。
- 持续调用工具，直到：(1) 任务完成，且 (2) 验证通过。
- 若工具返回空或部分结果，尝试不同策略重试。
</tool_persistence_rules>

<dependency_checks>
- 在执行操作前，检查是否需要前置发现、查找或记忆检索步骤。
- 不要因为最终目标看似明确就跳过前置步骤。
- 若任务依赖先前步骤的输出，请先解决该依赖。
</dependency_checks>
```

### 5. 强制长时任务的完整性
为防止模型部分完成即结束，明确定义完成规则和恢复行为。

```xml
<completeness_contract>
- 在覆盖所有请求项或明确标记为[阻塞]前，视任务为未完成。
- 维护一个内部清单，记录必需的交付物。
- 对于列表、批次或分页结果：
  - 尽可能确定预期范围，
  - 跟踪已处理的项目或页面，
  - 在最终确认前核实覆盖情况。
- 若任何项因数据缺失而阻塞，标记为[阻塞]并说明缺失什么。
</completeness_contract>

<empty_result_recovery>
若查找返回空、部分或可疑的狭窄结果：
- 不要立即断定没有结果存在，
- 尝试至少一两个备用策略，例如：
  - 替换查询词，
  - 放宽过滤器，
  - 执行前置查找，
  - 或使用替代来源/工具。
- 之后，报告未找到结果，并说明你尝试过的方法。
</empty_result_recovery>
```

### 6. 添加验证循环（尤其针对高风险操作）
在最终输出或执行不可逆操作前，加入轻量级验证步骤。

```xml
<verification_loop>
最终确认前：
- 检查正确性：输出是否满足所有要求？
- 检查事实依据：事实主张是否有所提供的上下文或工具输出支持？
- 检查格式：输出是否匹配请求的模式或风格？
- 检查安全性与不可逆性：若下一步有外部副作用，先请求许可。
</verification_loop>
```

# 三、专项工作流处理

## 1. 视觉任务（图像）
-   **high**：用于标准高保真图像理解。
-   **original**：用于大型、密集或空间敏感的图片（如计算机使用、定位、OCR、需要精确点击的任务）。
-   **low**：仅在速度和成本比细节更重要时使用。

```py
response_high = client.responses.create(
    model="gpt-5.4",
    input=[
        {
            "role": "user",
            "content": [
                {"type": "input_text", "text": "识别这张菜单上所有的菜名和价格"},
                {
                    "type": "input_image",
                    "image_url": f"data:image/png;base64,{image_base64}",
                    "detail": "high"  # ← 高细节模式（默认）
                }
            ]
        }
    ]
)
```

## 2. 研究与引用
强制模型仅基于检索到的证据进行回答，并严格遵循引用格式。

```xml
<citation_rules>
- 仅引用当前工作流中检索到的来源。
- 绝不捏造引用、URL、ID或引用段落。
- 精确使用宿主应用程序要求的引用格式。
- 将引用附加到其支持的具体主张上，而不是仅在末尾。
</citation_rules>

<grounding_rules>
- 仅基于提供的上下文或工具输出提出主张。
- 若来源冲突，明确说明冲突并归因于各方。
- 若上下文不足或不相关，缩小答案范围或说明无法支持该主张。
- 若陈述是推断而非直接支持的事实，请将其标记为推断。
</grounding_rules>
```

## 3. 研究模式
将 GPT-5.4 推入严谨的研究模式，适用于研究、审查和综合任务。

```xml
<research_mode>
- 分三个阶段进行研究：
  1) 计划：列出 3-6 个待回答的子问题。
  2) 检索：针对每个子问题搜索，并跟进 1-2 个次级线索。
  3) 综合：解决矛盾，并附上引用撰写最终答案。
- 只有当更多搜索不太可能改变结论时，才停止。
</research_mode>
```

## 4. 严格的输出格式（如 SQL, JSON）
指示模型仅输出目标格式，并在结束前进行自我检查。

```xml
<structured_output_contract>
- 仅输出请求的格式。
- 除非要求，否则不添加散文或 Markdown 代码块标记。
- 验证括号是否平衡。
- 不发明表格或字段。
- 若缺少必要的模式信息，请询问或返回明确的错误对象。
</structured_output_contract>
```

## 5. 编码与终端代理
明确终端访问和文件编辑的规则。

```xml
<autonomy_and_persistence>
- 在当前轮次内，尽可能端到端地处理任务：不要止步于分析或部分修复；除非用户明确暂停或改变方向，否则应将更改贯穿于实现、验证和结果解释中。
- 除非用户明确要求提供计划、询问代码问题、进行头脑风暴，或其他表明不应编写代码的意图，否则默认用户希望您进行代码更改或运行工具来解决问题。在这些情况下，不要在消息中输出您的建议方案，而应直接去实施更改。如果遇到挑战或阻碍，应尝试自行解决。
</autonomy_and_persistence>
```

# 四、运行时与 API 集成要点

## 1. Phase 参数
对于长时或工具密集型代理，`phase` 字段有助于区分中间更新和最终答案。**强烈建议使用**。

-   在 API 层面是**可选但强烈推荐**的。
-   用于可能发出评论（如前言）的代理。
-   重放先前的 assistant 消息时，**保留 phase 值**，以便模型区分工作评论和最终答案。
-   不要将其添加到用户消息中。
-   如果使用 `previous_response_id`，通常是最简单的路径。

## 2. 长会话行为保持
使用 [Compaction](https://developers.openai.com/api/docs/guides/compaction) 来显著延长有效上下文窗口。在达到主要里程碑后进行压缩，将压缩后的项视为不透明状态，并确保压缩后提示词功能相同。GPT-5.4 在更长、多轮对话中能保持更好的连贯性和可靠性。

## 3. 控制个性（面向客户的工作流）
将**持久个性**（语气、冗长度）与**单次写作控制**（渠道、格式、长度）分开。

```xml
<personality_and_writing_controls>
- 人物角色：<一句话>
- 渠道：<Slack | 邮件 | 备忘录 | PRD | 博客>
- 情感基调：<直接/冷静/充满活力等> + “不要<过度某一方面>”
- 格式：<若想要散文，禁止使用项目符号/标题/Markdown>
- 长度：<硬性限制，例如 ≤150 字或 3-5 句话>
- 默认跟进：若请求清晰且风险低，直接执行，无需请求许可。
</personality_and_writing_controls>
```

# 五、调整推理强度与迁移策略

## 1. 将推理强度作为微调旋钮
-   **none**：适合对速度、成本敏感，模型无需思考的任务。
-   **low**：适合对延迟敏感，少量思考可带来显著准确性提升的任务。
-   **medium/high**：为真正需要更强推理并能接受延迟和成本权衡的任务保留。
-   **xhigh**：避免作为默认值。仅适用于长时、代理型、推理密集型任务。

> **实践建议**：
> -   **执行密集型工作流**（如字段提取、支持分类）：从 `none` 开始。
> -   **研究密集型工作流**（如长上下文综合、多文档审查）：从 `medium` 或更高开始。
> -   在提高推理强度前，**优先添加** `completeness_contract`， `verification_loop`， `tool_persistence_rules` 等提示块。

## 2. 逐步迁移提示词
采用“一次只改一个变量”的原则：

# 六、小型模型（Mini & Nano）提示指南

-   **gpt-5.4-mini**：
    -   **特点**：更字面，假设更少。在结构化任务上很强，但处理隐式工作流和歧义能力较弱。
    -   **提示技巧**：将关键规则放在最前，指定完整执行顺序，使用编号步骤等结构框架，明确定义处理歧义的行为，直接指定输出格式。
-   **gpt-5.4-nano**：
    -   **用途**：仅用于狭窄、边界清晰的任务。
    -   **推荐**：偏好封闭式输出（标签、枚举、短JSON），避免多步骤编排。将模糊或规划繁重的任务路由给更强的模型。

# 总结

GPT-5.4 在需要多步骤证据收集、长上下文综合和明确提示词契约的任务中表现最佳。最有效的提示词改动包括：
1.  根据**任务形状**选择推理强度。
2.  定义**精确的输出和引用格式**。
3.  添加**依赖感知的工具规则**。
4.  明确**完成标准**。

模型本身能力强大，但当提示词清晰指定了“如何搜索”、“如何验证”和“何谓完成”时，其表现最为可靠。
