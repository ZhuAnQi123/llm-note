### 核心考察问题与深挖路线

#### 6.1 推理引擎与 PagedAttention 原理 (Inference Engines & PagedAttention)

* **面试官提问：**
> “在 LLM 推理服务中，为什么传统的 PyTorch 原生部署很难支持高并发访问？vLLM 引擎核心引入的 PagedAttention 机制是如何解决显存碎片化问题的？”


* **追问路线：**
* *追问 1（并发技术）：* 什么是 Continuous Batching（连续批处理 / Dynamic Batching）？它与传统固定 Batching 相比优势在哪里？
* *追问 2（关键指标）：* 在评估线上 LLM 推理服务性能时，我们会关注 TTFT（Time to First Token）和 TPOT（Time Per Output Token），这两个指标分别受什么因素影响？


* **参考答案/关键考点：**
* **PagedAttention 原理：** 借鉴了操作系统中的虚拟内存与分页机制。传统 Transformer 在 Autoregressive 阶段需要预先分配连续的显存空间来存储 KV Cache，导致大量的显存碎片（Memory Fragmentation）和内部浪费；PagedAttention 将 KV Cache 划分为固定大小的“块（Blocks）”，允许在非连续的显存空间中存储 Block，大幅提高了显存利用率（利用率提升至 96%+），使得并发 Batch Size 提升数倍。
* **Continuous Batching：** 传统 Batching 需要等待 Batch 内最长的请求生成完毕才能结束；Continuous Batching 在 Token 级别做调度，某个请求生成完毕（遇到 EOS）后立即将新请求插入 Batch 中，极大地提高了 GPU 利用率。
* **TTFT vs TPOT：**
* **TTFT (首 Token 延迟):** 主要由 Prefill 阶段（并行处理 Prompt 序列）决定，属于 **Compute-bound（计算密集型）**；
* **TPOT (后续 Token 生成速度):** 主要由 Decode 阶段（逐个自回归生成）决定，频繁从显存读取 KV Cache，属于 **Memory-bandwidth-bound（显存带宽密集型）**。

#### 6.2 量化技术与性能权衡 (Quantization)

* **面试官提问：**
> “为了在有限的显存上部署更大参数量的模型，我们常使用量化技术（如 AWQ, GPTQ, GGUF/bitsandbytes）。请说明 INT4 / INT8 量化的基本原理，以及不同量化算法对精度与推理延迟的影响。”


* **追问路线：**
* *追问：* 既然 INT4 量化能将显存开销降低 75%，那么在所有业务场景下都应该直接上 INT4 吗？量化在什么场景下容易失效或导致模型性能崩溃？


* **参考答案/关键考点：**
* **量化原理与变体：** 将 FP16/BF16 的浮点数映射到 INT8/INT4 整数范围（Scale & Zero-point）。
* **GPTQ:** 基于二次量化误差最小化，属于离线 PTQ（训练后量化）。
* **AWQ (Activation-aware Weight Quantization):** 考虑到激活值中只有 1% 的重要通道（Outliers）决定了性能，仅保护这 1% 的重要权重不被严重量化，精度损失比 GPTQ 更低。
* **GGUF (llama.cpp):** 适合 CPU/边缘端推断的二进制格式。


* **适用边界：** 复杂的代码生成、数学推理、高度敏感的结构化 JSON 输出等场景对量化极其敏感，INT4 量化可能会导致逻辑退化（Perplexity 飙升）；而对于简单的总结、分类、闲聊场景，AWQ INT4 几乎无损。

#### 6.3 提示词缓存与成本控制 (Prompt Caching & Cost Reduction)

* **面试官提问：**
> “如果我们的业务系统（如超长 System Prompt 的 Agent 或复杂 RAG）每天处理数百万次 Token 调用，有哪些工程手段可以把 API 成本或自建集群成本降低 50% 以上？”


* **追问路线：**
* *追问：* 提示词缓存（Prompt Caching）的原理是什么？它要求 Prompt 的结构遵循什么规则才能命中缓存？


* **参考答案/关键考点：**
* **降低成本的手段：**
1. **Prompt Caching（前缀缓存）：** 模型提供商（如 Anthropic/OpenAI）或开源引擎（如 vLLM Chunked Prefill）会缓存重复的前缀 Token 对应的 KV Cache。**要求固定不变的内容（如 System Prompt, Few-Shot 样例, 检索出的文档）必须严格放在 Prompt 前部**，动态变动的内容（如 User Query）放在最后。
2. **语义缓存（Semantic Cache）：** 使用 GPTCache / Redis Vector，对用户输入计算向量，命中历史相似问题（相似度 $> 0.95$）直接返回缓存答案，彻底避开大模型调用。
3. **大小模型路由（Model Cascading / Routing）：** 前置轻量级分类小模型（或 8B 模型）判断请求难度，简单请求交给小模型，极难请求才路由至顶级大模型。
