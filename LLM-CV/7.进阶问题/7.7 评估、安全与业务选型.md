

# 模块七：评估、安全与业务选型 (Evaluation, Guardrails & System Design)

> **面试官视角与定位：**
> 区分“只会闷头写代码的技术员”与“具备全局思维、能对最终业务结果与安全合规负责的 AI 团队 Leader/架构师”。核心考察候选人如何解决 **“大模型评估难”**、**“安全合规护栏”** 以及 **“复杂业务系统的技术选型”** 等综合痛点。

### 核心考察问题与深挖路线

#### 7.1 大模型评估体系与 LLM-as-a-Judge (LLM Evaluation)

* **面试官提问：**
> “在传统软件中，我们可以用单元测试做断言；但在 LLM 应用中，输出具有随机性和主观性。你在项目中是如何建立自动化评测集（Eval Suite）的？如何解决 LLM-as-a-Judge（大模型作为裁判）时带来的偏见问题？”


* **追问路线：**
* *追问：* 针对 RAG 系统的专项评估，你会监控哪些核心指标？（如 Ragas 框架中的 Faithfulness, Answer Relevance, Context Recall, Context Precision）


* **参考答案/关键考点：**
* **LLM-as-a-Judge 偏见与防御：**
* *常见偏见：* **位置偏见（Position Bias）**（倾向于选第一个）、**冗长偏见（Verbosity Bias）**（回答越长打分越高）、**自我偏见（Self-Enhancement Bias）**（倾向于给自己的模型打高分）。
* *防御手段：* 交换两个待评估回答的位置跑两次并取平均（Position Swapping）；在 Evaluation Prompt 中强制要求先输出“简要理由”再打分；限制输出长度因子影响；使用针对打分微调的小裁判模型。


* **RAG 四维评估 (Ragas Metrics)：**
* **忠实度 (Faithfulness):** 生成的答案是否完全基于检索到的 Context，无凭空捏造（测幻觉）；
* **答案相关性 (Answer Relevance):** 答案是否精准回答了 User Query；
* **上下文召回率 (Context Recall):** 检索到的 Context 是否包含了回答问题所需的全部关键信息；
* **上下文精确度 (Context Precision):** 检索出的 Context 中噪声比例是否极低（相关块是否排在前面）。

#### 7.2 安全护栏与事实性网关 (Guardrails & Fact-Checking)

* **面试官提问：**
> “假设你的 LLM 系统上线后，出现了政治敏感言论、竞争对手恶意抹黑或严重的幻觉错误。你在系统架构的输入和输出端各设计了怎样的 Guardrails（安全护栏）与事实性校验网关？”


* **追问路线：**
* *追问：* 如果在输出端挂载一层安全检测大模型，会导致整体响应延迟（Latency）增加一倍，你在工程层面如何做性能平衡？


* **参考答案/关键考点：**
* **双向安全护栏架构：**
* **输入端 (Input Guardrails):** 快速规则过滤（敏感词黑名单） + 轻量级分类模型（如 Llama Guard / NeMo Guardrails），阻断 Jailbreak 与违规 Query。
* **输出端 (Output Guardrails):** 基于 NLI（自然语言推理）轻量级模型做 Entailment（蕴含）校验；若判定生成结果与检索 Source 不蕴含，直接拦截。


* **延迟优化：** 使用**流式异步并行校验（Async Streaming Verification）** —— 在向前端 Stream 吐 Token 的同时，后台以 Sentence / Chunk 为单位做并行安全审计，一旦发现违规立即中断流传输并抛出系统错误。

#### 7.3 端到端系统设计与综合选型 (System Design / Scenario Question)

* **面试官提问（终极系统设计题）：**
> “请设计一套企业级‘智能法律合规助手’系统。要求：能解析 500 页包含复杂条款和对比表格的 PDF；支持针对特定条款的多轮问答与修改建议；必须严格输出 JSON 格式供前端渲染比对框；整体 API 响应时延（TTFT）控制在 1 秒以内，且保证敏感数据绝不外泄。请给出你的完整技术架构选型，并说明成本与延迟控制方案。”


* **追问路线：**
* *追问：* 面对这个需求，你会选择“完全自建开源模型微调”，还是“RAG + 商业大模型 API”？决策依据是什么？


* **参考答案/关键考点（考察架构综合能力）：**
* **技术选型决策链：**
* **数据安全/敏感性：** 涉及核心法律条款与敏感数据，优先考虑**本地化私有部署开源模型**（如 Qwen2.5-72B / Llama-3.3-70B）。
* **文档解析：** 采用多模态文档解析（ColPali / Visual-RAG）或版面分析模型提取表格与排版结构。
* **检索架构：** “BM25 + Dense Vector 混合检索 + Cross-Encoder Reranker”，配合 Parent-Child 分块（保证条款上下文完整性）。
* **格式确定性：** 使用 vLLM 部署本地模型，开启 Grammar-guided Decoding (GBNF Schema) 强行约束 JSON 输出。
* **延迟与性能控制：** vLLM + PagedAttention；对固定系统 Prompt 启用 Prefix Caching；将文档摘要与向量提前离线计算索引。

### 📝 评分标准 (Rubric)

| 判定等级 | 候选人表现特征 |
| --- | --- |
| **Junior (不合格/初级)** | 评测仅靠“人工看一眼觉得好不好”；不知道什么是 Position Bias；对系统选型缺乏成本与安全概念。 |
| **Middle (合格)** | 能使用 Ragas 等框架做评估；了解 LLM-as-a-Judge 的常见偏见；能设计基本的输入/输出敏感词过滤；能说明 Prompt vs RAG vs 微调的选择标准。 |
| **Senior / Architect (资深/架构师)** | 具备完整自动化 Eval 管道设计能力；掌握流式异步安全护栏设计，能在零感延迟下做输出拦截；能够针对复杂的真实企业场景，给出兼顾**安全性（私有部署）、确定性（Grammar 约束）、性能（vLLM/Prefix Caching）与成本（ROI）**的顶级系统设计方案。 |
