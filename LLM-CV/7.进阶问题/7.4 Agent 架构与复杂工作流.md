
# 模块四：Agent 架构与复杂工作流 (Agentic Workflow & LangGraph)

> **面试官视角与定位：**
> 区分“只会写简单的 `while` 循环调 API 的开发”与“能够设计高健壮性、可运行在生产环境的复杂 Agent 系统架构师”。核心考察候选人对**状态机机制**、**人工干预（HITL）**、**工具调用的死循环/容错**以及**多 Agent 协同**的工程掌控力。

---

### 核心考察问题与深挖路线

#### 4.1 Agent 推理机制与从 ReAct 到 DAG 的演进 (Agentic Workflows)

* **面试官提问：**
> “标准的 ReAct (Reason + Action) 范式在处理步数较多的复杂业务时，非常容易陷入‘逻辑死循环’或‘跑偏（Drift）’。你在工程上是如何约束和引导 Agent 的行为轨迹的？”


* **追问路线：**
* *追问 1（状态图）：* 为什么现在业界（如 LangGraph, AutoGen, LlamaIndex Workflows）越来越倾向于使用“状态图（State Graph / DAG）”而非纯自治的 Loop 来构建企业级 Agent？
* *追问 2（Memory）：* Agent 的长期记忆（Long-term Memory）和短期记忆（Short-term Memory）是如何管理的？当 Context Window 爆满时，如何保证不丢失关键决策状态？


* **参考答案/关键考点：**
* **ReAct 缺陷与 DAG 约束：** 纯自回归 Loop 缺乏硬性流程管控；资深架构师会使用**有向无环图 (DAG) / 状态机**将任务拆解为确定性节点（Node）与条件路由（Conditional Edge），让 LLM 仅在节点内部做决策，实现“硬编排 + 软智能”。
* **Memory 工程：**
* 短期记忆：对话 Message History、当前 Task State 变量控制；
* 长期记忆：提取 User Profile、历史 Task Summary 写入向量库或 KV 存储；
* Context 清洗策略：滑动窗口（Sliding Window）、Token 压缩算法（如 LongLLMLingua）或周期性状态 Summarization。





---

#### 4.2 LangGraph 与复杂状态管控 (LangGraph & State Management)

* **面试官提问：**
> “请结合 LangGraph 的核心概念，说明它是如何实现‘可暂停、可回滚、人工干预（Human-in-the-Loop）’的？其底层的数据状态（State）是如何持久化与传递的？”


* **追问路线：**
* *追问 1（人工干预）：* 如果 Agent 即将执行一个危险动作（如删库、高额转账、发送外部邮件），系统如何在节点处挂起（Interrupt）、等待管理员在前端点击“批准/修改”后再恢复运行？
* *追问 2（容错降级）：* 当某个工具（Tool）调用超时或返回 500 错误时，Agent 架构应该如何设计自愈与降级机制？


* **参考答案/关键考点：**
* **LangGraph 持久化与状态机：** 基于 Checkpointer（如 Redis, Postgres）在每个节点执行前后对 State 做 Snapshot（快照）。
* **Human-in-the-Loop (HITL)：** 利用 `interrupt_before` 或 `interrupt_after` 设置断点，暂停图的执行；前端修改 State 后通过 `update_state` 注入新的指令或参数，随后继续（Resume）执行图。
* **容错与自愈：** 节点级别的 Retry Policies；在 State 中记录 `error_count`，若连续失败超过上限，通过条件边路由至 Fallback 节点（如人工客服或降级为规则回复）。



---

#### 4.3 多 Agent 协同与 SOP 落地 (Multi-Agent Systems)

* **面试官提问：**
> “在什么场景下应该使用多 Agent 架构（Multi-Agent）而不是单个强大的 Agent？多 Agent 之间常见的通信模式（如 Supervisor 模式、 Hierarchical 模式、 Peer-to-Peer 模式）各有什么优缺点？”


* **追问路线：**
* *追问：* 多 Agent 系统在生产环境中往往 Token 开销巨大且延迟极高（TTFT / 响应时间翻倍），你在工程上做过哪些性能与成本优化？


* **参考答案/关键考点：**
* **单 Agent vs 多 Agent：** 任务上下文过长、工具过多导致单一 Prompt 无法承载（Tool Choice 准确率急剧下降）时，必须按角色（Role / SOP）拆分为专业化小 Agent。
* **协同架构：**
* *Supervisor (主控模式):* 一个路由 Agent 负责任务分发与结果汇总；
* *Hierarchical (层级模式):* 树状管理架构，适合大型项目拆解；


* **性能优化：** 并行节点执行（Parallel Execution）；小 Agent 尽量改用参数量较小、成本极低的基座模型（如 8B / 70B 模型配合专用 System Prompt）；对不依赖推理的中间节点采用标准 Python 代码节点替换。
