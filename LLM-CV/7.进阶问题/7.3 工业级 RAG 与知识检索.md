# 模块三：工业级 RAG 与知识检索 (Advanced RAG & GraphRAG)

> **面试官视角与定位：**
> 考察候选人对“非结构化数据处理”、“向量/混合检索算法”以及“ GraphRAG / 多模态 RAG 等前沿技术”的掌握程度。核心看候选人能否解决**“查不准”**、**“查不全”**以及**“全局总结类问题回答差”**等真实落地痛点。

---

### 核心考察问题与深挖路线

#### 3.1 数据预处理与分块策略 (Data Ingestion & Chunking)

* **面试官提问：**
> “在搭建企业级 RAG 系统时，文档解析往往是‘第一道死门’（尤其是带复杂表格、页眉页脚、多栏排版的 100 页 PDF ）。你在实际项目中是如何做数据清洗与分块（Chunking）的？如何避免分块将完整的上下文切断？”


* **追问路线：**
* *追问 1（语义分块）：* 除了固定长度（Fixed-size）切片，你了解或使用过哪些高级分块策略？（如 Semantic Chunking, Parent-Child Chunking, Sentence Window）
* *追问 2（多模态解析）：* 对于文档中的架构图、流程图或大表格，传统 OCR 解析后丢给 Embedding 模型效果极差，你怎么解决？


* **参考答案/关键考点：**
* **高级 Chunking 机制：**
* **Parent-Child / Small-to-Big Chunking：** 用小 Chunk（如句子级）做精确向量检索，检索命中后将对应的 Parent Large Chunk（如段落/章节）喂给 LLM 提供完整上下文。
* **Semantic Chunking：** 基于语义相似度（如相邻句子的 Embedding 距离突变点）动态划分切片边界。


* **复杂文档/表格/图表解析：**
* 表格转 Markdown / HTML 格式，并生成 Table Summary 进行向量索引；
* 利用 Visual-RAG 或多模态文档解析模型（如 Unstructured / LayoutLM / ColPali）直接基于视觉布局做 Feature / Text 提取。





---

#### 3.2 混合检索与重排机制 (Hybrid Search & Reranking)

* **面试官提问：**
> “为什么单纯靠向量检索（Dense Retrieval）在企业落地中经常失效（比如搜索特定型号产品名、错别字或专有名词）？你是如何搭建‘混合检索 + 重排’架构的？”


* **追问路线：**
* *追问 1（重排原理）：* Cross-Encoder (Reranker) 和 Bi-Encoder (Embedding) 的底层计算机制有何不同？为什么重排模型准确度更高但不能直接用于海量库全量检索？
* *追问 2（查询转换）：* 如果用户的 Query 表达模糊或过于简短，你在 Retrieval 前做了哪些预处理？（如 Query Rewriting, Hypothetical Document Embeddings - HyDE）


* **参考答案/关键考点：**
* **混合检索架构：** **BM25 / Sparse Vector (稀疏向量)** 负责精确关键字、专有名词、型号的召回 + **Dense Vector (稠密向量)** 负责语义泛化召回，两者使用 **RRF (Reciprocal Rank Fusion)** 进行得分融合。
* **Reranker 底层区别：**
* *Bi-Encoder (Embedding):* Query 和 Document 分别独立计算向量，相似度用点积/余弦计算，速度极快（可利用 HNSW 索引），但忽略了词与词之间的交叉注意力。
* *Cross-Encoder (Reranker):* 将 Query 和 Document 拼接后一起输入 Transformer，利用 Full Attention 层充分计算语义相关性，精度高但计算开销大，故只用于 Top-K（如前 50 个）的精刷。





---

#### 3.3 GraphRAG 与全局关系检索 (GraphRAG)

* **面试官提问：**
> “传统 RAG 面对‘总结这本手册中所有的安全合规风险’这种全局归纳型（Global Query）问题时表现非常糟糕，为什么？GraphRAG（知识图谱增强 RAG）是如何解决这一问题的？”


* **追问路线：**
* *追问：* 微软开源的 GraphRAG 核心流程包含哪几个步骤？它在实际落地的成本（Token 消耗）和实时性上有什么致命缺点？你会在什么场景下选择用它？


* **参考答案/关键考点：**
* **GraphRAG 核心流程：** 文本切片 $\to$ LLM 抽取实体与关系（Entities & Relationships）$\to$ 构建知识图谱 $\to$ **社区发现算法（Community Detection, 如 Leiden 算法）** $\to$ LLM 生成不同层级社区的摘要（Community Summaries）$\to$ 检索时在社区摘要层级做 Map-Reduce。
* **适用场景与代价：** 适用于长文档库的全局提炼、关系洞察；缺点是索引构建极度昂贵（耗费大量 LLM Token 抽实体/做摘要），不适合实时频繁变更的动态数据。

### 📝 评分标准 (Rubric)

| 判定等级 | 候选人表现特征 |
| --- | --- |
| **Junior (不合格/初级)** | 认为 RAG 就是“把 PDF 传进 LangChain 然后转向量存 Chroma”；不知道 BM25 与向量检索的区别；没听说过 Reranker。 |
| **Middle (合格)** | 懂得 Parent-Child / Semantic Chunking 策略；落地过“BM25 + 向量检索 + Reranker”三段式流水线；能说明 Query 重写的作用。 |
| **Senior / Architect (资深/架构师)** | 清楚 Bi-Encoder 与 Cross-Encoder 的计算复杂度与矩阵计算差异；具备多模态文档/复杂表格解析方案；理解 GraphRAG / Community Detection 机制，且能从 ROI（成本/延时）角度评估技术选型。 |
