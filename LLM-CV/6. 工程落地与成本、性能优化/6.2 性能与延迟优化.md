# 6.2 性能与延迟优化 (Latency Optimization)

## Q8：在流式传输（Streaming）中，前端和后端是如何配合处理偶发性断线、用户主动终止的？

**🎯 核心考点：SSE 协议，ClientDisconnected 捕获，generator.aclose()。**

**参考答案：**
流式传输通常采用 **SSE（Server-Sent Events）** 协议。高并发线上环境的配合与资源释放机制如下：

1. **前端终止与断开感知：** 当用户在界面上点击“停止生成”或直接关闭标签页时，前端会主动中断这个 HTTP 连接。
2. **后端异步捕捉与熔断（关键工程细节）：**
* 后端（如使用 FastAPI 异步生成器 `async def event_generator()`）必须能够捕获客户端连接断开的异常信号（如 `asyncio.CancelledError` 或框架自带的 `ClientDisconnected`）。
* **强行释放算力：** 一旦捕获该信号，后端代码必须**立刻显式触发 `generator.aclose()` 截断机制**，终止与下游推理集群（如 vLLM）或第三方 API（如 OpenAI）的流连接。
* *避坑：* 如果不写这个捕获和中断，虽然前端断开了，大模型在服务端依然会把剩下的上千个 Token 全部生成完，这会引发**严重的线上算力被无效空耗和 Token 资费的无意义浪费**。

## Q9：请通俗解释大模型提示词缓存（Prompt Caching）的底层原理。它在哪些具体的业务场景下能帮你省下大笔开销？

**🎯 核心考点：KV Cache 的复用，Prefill（预填充）阶段的算力节省。**

**参考答案：**

1. **底层原理：**
在大模型推理的 **Prefill（预填充）阶段**，模型需要将输入的所有 Prompt 文本通过自注意力机制计算出对应的 **KV Cache（键值缓存）**，这个阶段的计算复杂度与 Prompt 长度呈平方级关系。
* **Prompt Caching** 的思想是：如果多名用户输入的 Prompt 中，**有一大段前缀（Prefix）是完全一模一样的**（例如很长的系统提示词、或者同一本企业规章制度），推理引擎（如 vLLM 或 DeepSeek/OpenAI 服务端）就会把这段前缀算好的 KV Cache **直接缓存在显存或内存中**。下一个请求进来时，直接复用这段缓存，跳过繁重的 Prefill 计算。


2. **核心省钱与加速场景：**
* **长文本多轮对话：** 随着对话轮数增加，前面的历史聊天记录越来越长。开启 Prompt Caching 后，模型每次只需要计算最新一轮对话的几百个 Token，前面的几万字历史记录直接走缓存。**速度提升 2x~5x，API 费用通常能拿到 2 折~5 折的巨大优惠（如 DeepSeek 的缓存命中价格极便宜）**。
* **固定知识库 RAG：** 比如针对同一本 20 万字的产品手册进行高频客服问答。

## Q10：什么是语义缓存（Semantic Cache）？它和传统 Redis 的字符串 Key 缓存有什么本质区别？

**🎯 核心考点：向量相似度匹配缓存，阈值截断防“失真”。**

**参考答案：**

1. **与传统缓存的本质区别：**
* **传统缓存（如 Redis String Cache）：** 要求 Key 必须**完全一致**。如果用户 A 搜 `"怎么改密码"`，用户 B 搜 `"如何修改密码"`，哪怕语义完全一样，传统缓存也会因为字符串对不上而无情穿透，被迫让大模型重新生成一遍。
* **语义缓存（Semantic Cache，如 GPTCache）：** 它的 Key **不是字符串，而是问题的 Embedding 向量**。


2. **工作原理与工程设计：**
* 当用户输入一个新问题时，系统先计算其 Embedding，然后去支持向量检索的缓存库里进行 **Top-1 相似度检索**。
* 系统会设置一个非常严格的**相似度截断阈值 $\tau$（如余弦相似度 $> 0.96$）**。
* **命中缓存（Cache Hit）：** 如果库里存在一条历史问题与当前提问的相似度大于 $\tau$，判定语义完全等价，系统**绕过大模型，直接把那条历史缓存的答案秒级返回给用户**。


3. **收益与风险平衡：**
* *收益：* 实现毫秒级响应（Latency $\to 0$），且大模型 Token 成本直接归零。
* *风险（必须注意）：* 阈值 $\tau$ 如果设得太低（如 0.85），会导致严重的**答非所问（缓存污染）**。例如把 `"我想买 iPhone 15"` 误匹配成了 `"我想买 iPhone 14"` 的缓存答案。因此语义缓存只适合用于高频、标准、长尾效应明显的 FAQ 场景。
