# 5.3 线上监控与可观测性 (LLMOps)

## Q5：在大模型线上生产环境中，你如何设计指标监控看板？对于无标准答案的流式场景，你会追踪哪些指标？

**🎯 核心考点：性能、成本、质量的多维实时监控。**

**参考答案：**
线上看板必须实现性能（Performance）、成本（Cost）、质量（Quality）的三维闭环监控：

1. **性能指标（实时流）：**
* **TTFT（首字延迟）：** 监控首个 Token 吐出时间，卡死在 1.5 秒内，直接影响用户体验。
* **TPOT（单字生成延迟）：** 监控流式打字机的速度，评估后端集群或第三方 API 是否限流。


2. **成本指标（财务线）：**
* **累计 Token 消耗与 TPM（每分钟 Token 数）：** 按用户、按业务线分类统计输入/输出 Token，配合 Redis 设置异常高频调用的**断路器/熔断器**，防止 Agent 死循环烧钱。


3. **质量指标（无监督自适应）：**
* **隐式行为指标：** 用户点赞/点踩率、文本复制率、连续点击“重新生成”的频率。
* **异步抽样忠实度：** 将 1% 的线上真实请求异步推入 Kafka，用轻量级模型自动跑 Ragas 忠实度检测，监测线上幻觉率的漂移。

## Q6：在工业界落地大模型可观测性（Observability）时，类似 LangSmith 或 Phoenix 这样的工具链核心解决了什么问题？它们是如何展现一个复杂 Agent 工作流的？

**🎯 核心考点：分布式链路追踪（Distributed Tracing），Span 与 Trace 的层级嵌套。**

**参考答案：**

1. **核心解决的痛点：**
在没有这些工具前，大模型后端是一张黑盒。一个 Agent 运行失败或响应变慢，你根本分不清是因为 Prompt 拼错了、向量库检索超时、还是大模型调用中间工具时吐出了坏 JSON。LangSmith 和 Phoenix 核心解决了**大模型复杂长链路下的透明化调试与全生命周期追踪问题**。
2. **如何展现复杂工作流（层级 Trace 架构）：**
* 它们基于 **OpenTelemetry 标准**。当用户发起一次复杂请求，系统会生成一个全局 `Trace ID`。
* 随着流程向下传导，每一次的子操作都会被封装成一个 **`Span`（跨度）**，并记录其耗时、输入、输出和元数据（Metadata）：
* 父 Span ──> `Agent_Run`（记录整个业务起止）
* 子 Span 1 ──> `Query_Rewriter`（大模型重写问题）
* 子 Span 2 ──> `Vector_Search`（检索向量库，可视化展现召回的各个网页和 score）
* 子 Span 3 ──> `LLM_Chain`（大模型真正吃进上下文进行推理，完整暴露当时的 Prompt 现场）




* 工程师在看板上可以通过“树状时间线图”清晰地看到哪一个步骤耗时最长、哪一步的 Prompt 导致了模型报错，甚至可以**一键把线上的坏用例（Bad-case）直接导出到 Playground 调优，实现快速闭环修复**。
