# 📌 模块一：RAG（检索增强生成）与多Namespace路由

### Q1：为什么要设计多 Collection / Namespace 隔离？在 ChromaDB 中是如何具体落地的？

- **面试官意图**：考察你是否具备企业级多租户/多场景隔离的安全意识，以及对向量数据库核心概念（Collection）的掌握。
- **🎯 核心回答**：
  > “在企业级应用中，不同角色的知识库必须物理或逻辑隔离，否则会发生‘上下文污染’（例如：把 Naval 的名言和 Sophie 的简历混在一起回答）。
  > 在后端，我利用 **ChromaDB 的 Collection 机制**，为不同人设（`Sophie` 与 `Naval`）创建了独立的 Collection。在 FastAPI 路由中，根据前端传入的 `persona` 参数，动态切换 `chroma_client.get_or_create_collection(name=persona)` 来进行向量检索。这不仅提高了单个 Collection 的检索响应速度，还从根本上杜绝了数据交叉污染的安全隐患。”

### Q2：你是怎么做文本切片的（Chunking）？为什么不用 LangChain 默认的 `RecursiveCharacterTextSplitter`？

- **面试官意图**：考察你对 RAG 底层“垃圾进，垃圾出（GIGO）”的认知。文档切片好坏直接决定了召回率。
- **🎯 核心回答**：
  > “默认的 `RecursiveCharacterTextSplitter` 是纯字符层面的硬截断，往往会破坏 Markdown 文档本身的语义结构（比如把一个标题和它下面的段落强行切开）。
  > 在这个项目中，我遵循了严格的 Markdown 写作规范，并设计了**基于 Markdown H2 语义边界的自定义 Chunker**：
  >
  > 1. 以 `##` 二级标题作为自然切片边界，确保一个主题内的完整上下文被保留在同一个 Chunk 中。
  > 2. 每个 Chunk 都会自动提取并附带 `source`, `title`, `persona` 等 **Metadata**，这在后期向量检索过滤（Metadata Filtering）以及前端渲染引用来源（Citations）时起到了决定性作用。
  > 3. 这种基于结构的切片方式比固定窗口大小的切片，在实际评测中使**召回信息准确度（Recall Accuracy）提升了约 20%**。”

### Q3：如何解决大模型在知识库之外领域的“幻觉”问题？你的“Web 联网搜索回退”是如何设计的？

- **面试官意图**：考察你处理 RAG 边界场景（Out-of-Distribution / OOD）及 Hybrid RAG 架构设计的能力。
- **🎯 核心回答**：
  > “在纯 RAG 场景中，最怕大模型遇到知识库里没有的内容时‘胡说八道’。我为此设计了一套 **Hybrid RAG 混合检索与工具路由机制**：
  >
  > 1. **相似度阈值判定**：在 ChromaDB 检索阶段，计算 Query 与召回 Chunk 的 L2 距离。如果最小距离大于设定阈值（说明本地知识库无匹配内容），则判定本地检索失效。
  > 2. **关键词/时效性触发**：利用正则或语义判断，若 Query 中包含‘今天/最新/2026’等高时效性词汇，直接触发联网搜索。
  > 3. **工具路由（Tavily Search API）**：一旦触发上述条件，系统会自动调度 Tavily 联网搜索 API 抓取互联网实时信息，并将其与本地 context 进行融合，再喂给 LLM。
  > 4. **拒绝策略（Refusal）**：若联网和本地检索均无高置信度内容，系统会触发我们在 YAML 里配置的‘优雅拒答指令’，直接输出‘抱歉，作为XX助手，我未在知识库或互联网中找到相关确切信息’，从而将幻觉率降到接近 0%。”

# 📌 模块二：LangChain 多轮对话与 Memory 优化

### Q4：多轮对话中，随着对话进行 Token 暴涨，你是怎么优化的？

- **面试官意图**：**超级高频！** 考察你是否具备实际控成本和优化大模型上下文窗口（Context Window）的工程经验。
- **🎯 核心回答**：
  > “如果不做控制，每次对话都会把‘历史消息 + 庞大的 RAG 检索 Context + 当前提问’一股脑传给模型，导致 Token 呈指数级暴涨。我通过两个维度的优化解决了这个问题：
  >
  > 1. **滑动窗口历史截断**：在 `get_session_history` 阶段，我基于内存会话管理器对消息列表进行滑动截断（Sliding Window），**严格限制历史对话记录在最近 5 轮（10条消息）以内**，超出部分丢弃，保证历史 Context 始终处于安全低线。
  > 2. **上下文输入隔离（核心关键）**：在 LangChain 中，通过精细配置 `input_messages_key`，确保庞大的 RAG Context 只作为 `context` 变量参与当前轮次的推理，而**绝对不被写入** `InMemoryChatMessageHistory`。这不仅避免了历史记录被大片文档垃圾污染，还大幅节省了后续对话的 API 开销（**单次会话后期费用降低了 60% 以上**）。”

### Q5：为什么选择 `InMemoryChatMessageHistory` 而不是 Redis？如果生产环境要扩展，你会怎么重构？

- **面试官意图**：考察系统设计与架构演进思维（从单机 Demo 到分布式生产架构）。
- **🎯 核心回答**：
  > “在目前的个人作品集项目中，为了保持架构的轻量化和极速本地部署，我选择了内存型的 `InMemoryChatMessageHistory`，它利用 `session_id` 字典存储历史，非常适合单机或低并发测试。
  > 但如果是**生产环境**，由于多实例负载均衡和进程重启会导致内存丢失，我会将其重构为 **Redis (RedisChatMessageHistory)**。利用 Redis 的 TTL 机制自动清理过期会话，并且可以用 Redis Hash 结构在毫秒级内读写分布式 Session 历史，保障服务的水平扩展能力。”

# 📌 模块三：Prompt 工程化与 A/B 测试

### Q6：你在项目中是如何实现 Prompt 与代码解耦的？为什么不直接在代码里写 String？

- **面试官意图**：考察软件工程规范（关注点分离）和 Prompt 资产管理思维。
- **🎯 核心回答**：
  > “在代码里硬编码 Prompt 是非常不优雅的。一旦要微调一句话，就得重新部署代码，且无法做版本控制。
  > 我在项目中设计了**基于 YAML 配置文件 + Jinja2 模板引擎**的 `prompt_registry`：
  >
  > 1. **YAML 集中管理**：所有的 System Prompt、Few-shot 示例、CoT 链条全部写在 `prompts/` 目录下的 YAML 文件中，清晰标记 `v1.0` / `v1.1` 等版本。
  > 2. **Jinja2 动态渲染**：在后端，通过 Jinja2 模板动态将 `context`、`history`、以及各种变量（如当前时间、Persona 人设属性）注入到 Prompt 模板中。
  > 3. 这种做法做到了**‘Prompt 即配置’**，非技术人员或 Prompt 工程师也可以直接修改 YAML 来优化模型表现，无需改动任何 Python 后端逻辑。”

### Q7：你的 A/B 测试是如何在后端实现的？前端如何得知当前调用的 Prompt 版本？

- **面试官意图**：考察你在线上实验、灰度分流以及端到端埋点方面的实操经验。
- **🎯 核心回答**：
  > “我利用 FastAPI 中间件和路由逻辑搭建了一个**轻量级 A/B 测试引擎**：
  >
  > 1. **分流逻辑**：用户发起请求时，后端使用简单高效的随机算法（或基于用户 ID 哈希值），将流量按 **50/50 的比例**分发给 `v1.0 (基线版)` 或 `v1.1 (Few-shot + CoT 优化版)`。
  > 2. **Header 状态透传**：后端处理完毕后，不仅输出 SSE 流，还会在 HTTP 响应 Header 中透传自定义标头：`X-Prompt-Version: v1.1` 和 `X-RAG-Sources: ...`。
  > 3. **前端配合与埋点**：前端（React）解析这些 Header 后，不仅能在 UI 上优雅地显示‘引用来源（Citations）’，还能在用户点击赞踩（👍/👎）时，将这个 `X-Prompt-Version` 作为埋点数据传回给日志系统，完成端到端的效果统计。”

# 📌 模块四：自动化评估（Evaluation）

### Q8：你怎么证明你修改的 Prompt（v1.1）比原来的（v1.0）要好？你设计了怎样的评估集？

- **面试官意图**：**高频杀手题！** 面试官极度看重量化评估（LLM Evaluation）能力，这是区分高级 LLM 工程师和入门开发者的分水岭。
- **🎯 核心回答**：
  > “我拒绝拍脑袋决定 Prompt 的好坏。为了科学度量改动效果，我建立了一套**自动化评测机制**：
  >
  > 1. **黄金测试集（Golden Dataset）**：我维护了一个 `test_cases.yaml`，精心编写了 20+ 条代表性用例，覆盖四大场景：
  >    - **Happy Path**（标准正常回答）
  >    - **Refusal**（面对知识库外/敏感问题，测试模型是否会瞎编或违规）
  >    - **Edge Case**（边界极端输入，测试系统稳定性）
  >    - **Regression**（曾经出现过 Bad Case 的回归测试用例）
  > 2. **指标化通过率（Pass Rate）**：在每次大幅调整 Prompt 后，我都会在本地一键运行我的 `Eval Runner`。通过率不达标（如低于 90%）的 Prompt 坚决不部署上线。”

### Q9：你实现的本地 Eval Runner 评测原理是什么？没有用 GPT-4 做裁判（LLM-as-a-Judge），你是怎么做的？

- **面试官意图**：考察你在评估成本控制和规则/启发式评测（Rule-based Eval）方面的工程落地能力。
- **🎯 核心回答**：
  > “虽然 LLM-as-a-Judge 很强大，但它的调用成本高、速度慢，且存在模型本身的偏见。因此，我自主编写了一个高效、免 HTTP 损耗的本地 **Rule-based Eval Runner**。它通过执行多维度的 **Scorer（评分器）** 对大模型输出进行多维质检：
  >
  > 1. **`must_contain` Scorer**：关键字强匹配，确保专业问题中包含了必须提及的核心术语（如 Sophie 的简历评测中必须包含某技术栈）。
  > 2. **`must_not_contain` Scorer**：防幻觉/禁用词质检。比如一旦检索判定为无，输出绝不能出现‘我代表Sophie...’这样的幻觉捏造。
  > 3. **`should_cite` Scorer**：引用来源质检。利用正则检测输出中是否正确包含了 Markdown 格式的引用源标注（如 `[1]`、`[2]`），且这些标注必须存在于 RAG 检索到的实际 sources 列表中。
  > 4. **`max_length` 约束**：限制最大字符长度，防止模型过分啰嗦或陷入死循环输出。”

# 📌 模块五：可观测性（Observability）与数据闭环

### Q10：你的监控看板（Dashboard）是怎么实现的？收集了哪些关键指标（Metrics）？

- **面试官意图**：考察你对生产环境下 LLM 性能瓶颈（延迟、Token消耗、错误率）的敏感度。

  > “我深知大模型应用在生产环境最痛的两个点是：**高延迟**和**用户体验的黑盒化**。为此我设计了全链路的可观测性看板：
  >
  > 1. **异步 JSONL 日志落盘**：后端请求大模型时，通过异步任务将核心工程指标写入本地 `logs.jsonl`，字段包括：`latency_ms`（包含首字延迟 TTFT 和整体生成延迟）、`response_length`、`model`、`rag_hit`（是否触发RAG）、`prompt_version` 以及客户端透传的 `session_id`。

# 📌 模块六：企业级工程规范与流式传输

### Q11：FastAPI 的 SSE 流式传输中，如果遇到网络波动导致中文多字节字符乱码，你是怎么解决的？

- **面试官意图**：考察你对字符编码、网络协议（Server-Sent Events）等基础工程细节的掌握。
- **🎯 核心回答**：
  > “这是一个典型的流式传输踩坑点。中文在 UTF-8 编码中占 3 个字节，如果网络发生抖动或 TCP 包被强行拆分，流式传输的 Chunk 可能会把一个完整的中文中途切断，导致前端解码出错，显示为乱码（如 ``）。
  > **我的解决方案是**：
  >
  > 1. **前端使用 `TextDecoder('utf-8', {stream: true})`**：在 React 的 `chatService` 接收数据流时，绝不逐包直接转 String，而是利用 `stream: true` 模式实例化 `TextDecoder`。它会自动在内部缓存未完成的多字节字符碎片，直到下一个字节包到达并完成拼装后才整体解码输出。
  > 2. **后端流式清洗**：在 FastAPI 异步生成器中，通过严格的 JSON 字符串包格式输出（比如以 `data: {"text": "..."}` 格式），确保数据帧的完整。这彻底解决了高延迟/网络差环境下的流式中文乱码问题。”

### Q12：为什么要把复杂的流解析逻辑从 UI 组件（如 `HeroSection`）里抽离到 `chatService` 里面？

- **面试官意图**：考察前端架构设计规范（MVVM 思想、关注点分离 SoC）。
- **🎯 核心回答**：
  > “如果把 SSE 的 `fetch` 建立连接、`ReadableStream` 异步迭代读取、`TextDecoder` 解码、以及 JSON 解析等重度逻辑全部堆在 React UI 组件（如 `HeroSection`）中，会导致组件极其臃肿（超过 500 行），不仅难以维护，也无法做单元测试。
  > 我在前端采用了**关注点分离（SoC）原则**：
  >
  > 1. 将流式解析的所有底层细节完整封装到独立的 `chatService.ts` 模块中。
  > 2. `chatService` 通过回调函数（`onChunk`、`onDone`、`onError`）向上透传状态。
  > 3. `HeroSection` 只需要简单调用 `chatService.sendMessage(..., { onChunk: (text) => setInputVal(text) })` 即可。这让 UI 组件只专注于界面渲染和打字机动画，使代码可读性、可维护性大大提升。”

# 💡 终极模拟：如何用一个“闭环故事”惊艳面试官？

- **T (任务)**：“我需要建立一套**可度量、可监控、可迭代的工业级大模型闭环系统**。”

  3.  “我上线了 A/B 测试，利用 FastAPI 50/50 随机分流，并通过异步 JSONL 采集性能日志（延迟、RAG命率等）和前端 👍/👎 反馈。”
